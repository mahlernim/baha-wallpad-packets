# 현관 / 일괄소등 노드 `1F 0F`

상위 문서: [README](../README.md)  
바로 가기: [프로토콜 개요](protocol-overview.md) | [하드웨어](hardware.md) | [거실 조명 `10 04`](lighting-node-10-04.md) | [난방 `40 90`](heater-node-40-90.md) | [참고 자료](../reference/README.md)

이 노드는 현관 쪽 일괄소등 스위치 브랜치를 연결했을 때 나타난다. 버스 내부에서의 정확한 구조적 역할은 아직 단정하지 않았지만, 상태 조회와 제어에 필요한 동작은 충분히 정리됐다.

## 조회 / 상태 응답

폴링 요청:

```text
02 1F 0F 01 00 13 00
```

응답 형식:

```text
02 1F 0F 81 01 <value> <chk> 00
```

안정 상태에서 관찰된 예:

```text
02 1F 0F 01 00 13 00 -> 02 1F 0F 81 01 0A 98 00
```

## 쓰기 형식

수용되는 쓰기 계열:

```text
02 1F 0F 02 01 <value> <xor_checksum> 00
```

관찰된 허용 값:

```text
08
09
0A
```

관찰된 거부 값:

```text
00
0B
0E
```

## 실사용 상태 매핑

| 값 | 의미 | 실사용 해석 |
| --- | --- | --- |
| `09` | OFF | 안정적인 사용자용 OFF |
| `0A` | ON | 안정적인 사용자용 ON |
| `08` | 일시 상태 / 비고정 상태 | 문서에는 남기되 일반 버튼 상태로 노출하지 않는 것이 안전 |

중요한 점:

- `08`은 허용되며 즉시 같은 값으로 응답한다.
- 안정 상태가 ON이면 다음 조회에서 다시 `0A`로 돌아온다.
- 직접 테스트에서는 `08`이 거실 조명 마스크를 곧바로 바꾸지는 않았다.

## 최종 사용 지침

실사용 기준으로는 다음처럼 정리하는 것이 맞다.

- OFF는 `09`
- ON은 `0A`
- `08`은 프로토콜상의 잔류 값 또는 일시 상태로 취급

## 남는 구조적 단서

아직 설명형 수준에 머무는 부분:

- `1F 0F`가 물리 현관 스위치 자체인지, 상위 컨트롤러인지, 상태 중계 노드인지
- 물리 버튼 입력이 `1F 0F` 쓰기를 직접 발생시키는지, 아니면 다른 컨트롤러를 통해 간접 반영되는지

이 점은 내부 토폴로지 설명에만 영향을 주며, 상태 해석과 제어 자체를 막지는 않는다.
